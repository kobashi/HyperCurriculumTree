# 同種プロジェクトの進め方チェックリスト

このメモは、HyperCurriculumTree 2026年度版 `Ver.1.0` の開発で発生した手戻りを踏まえ、次に同じような「資料から履修計画・要件判定・カリキュラムツリーを実装する」プロジェクトを効率よく進めるための手順をまとめたものです。

## 1. 最初に決めること

### 正本と補助資料を分ける

- 正本:
  最終的な科目名、系列、配当年次、矢印、卒業要件の根拠にする資料。
- 補助資料:
  正本だけでは不足する開講時期、実際の時間割、履修上の注意、教職課程、資格認定などを補う資料。
- 実装資料:
  正本と補助資料を読み解いて、アプリで扱いやすい形に再構造化した Markdown やデータ表。

この区分が曖昧なまま進めると、PDF、MD、アプリデータのどれを直せばよいか分からなくなり、同じ科目や前提関係を何度も修正することになる。

### 実装前に合意しておく判断

- アプリは実時間割を作るものか、履修計画を作るものか。
- 曜日・時限を扱うか、学年期だけ扱うか。
- 科目履修、集中講義、資格取得、単位認定を同じ選択 UI に含めるか。
- 履修判定用の前提関係と、ツリー表示用の接続を分けるか。
- 公式資料にない補完情報を入れる場合、暫定値として明記するか。
- UI 上の略称を使う場合、正式名をホバーなどで確認できるようにするか。

## 2. 資料分析フェーズ

### 収集チェック

- カリキュラムツリー PDF
- 教育課程表、履修の手引き、卒業要件資料
- 時間割資料
- 教職課程資料
- 資格取得による単位認定資料
- 他学科履修資料
- 科目番号やナンバリング資料
- 公式 Web ページの URL と取得日

### PDF 読み取りチェック

- PDF 上の全科目名を一覧化したか。
- ローマ数字、全角英字、記号、長音、旧字体などの表記揺れを整理したか。
- 見落としやすい小さな科目名を拡大して確認したか。
- 矢印の始点・終点を、見た目だけでなく資料上の配置から確認したか。
- 複数前提科目を持つ科目を別表に抜き出したか。
- 系列内の見やすさ用接続と、履修上の必須前提を区別したか。
- 配当年次なし、集中講義、資格認定など通常枠から外れる科目を別扱いにしたか。
- 教職課程のように「個別科目の前提」ではなく「半期群の積み上げ」として読むべきものを識別したか。

### MD 化チェック

- 公式資料から読んだ情報を、まず Markdown に落としてから実装したか。
- Markdown 内に、正本由来か補助資料由来かを書いたか。
- 実装に入る前に、MD と PDF の差分を確認したか。
- 実装後に手修正した内容を、アプリだけでなく MD に戻したか。
- 「担当教員からの補足」「暫定番号」「仮配置」など、公式資料と異なる情報に印を付けたか。

## 3. データ設計フェーズ

### 科目マスタ

科目ごとに、最低限次を持たせる。

- 一意 ID
- 正式科目名
- 表示名、略称
- 単位数
- カテゴリ
- コース
- 系列
- 配当年次
- 開講期
- CAP 上限内/上限外
- 教職必修
- コース必修
- 資格認定対象
- 公式番号または暫定番号
- 参照元資料

### 表記揺れ

- `normalizeName` のような正規化関数を早い段階で作る。
- 別名、旧表記、全角半角違いは aliases に集約する。
- 正式名称を決めたら、UI、MD、判定ロジック、ツリー表示の全てで同じ名称を使う。

### 前提関係と表示用接続

- 履修判定用:
  実際に履修順チェックに使う厳密な前提関係。
- ツリー表示用:
  公式ツリーの矢印、系列の流れ、半期群の積み上げを見せるための接続。

この 2 つを混ぜると、見やすさのための矢印が履修エラーになったり、逆に厳密な前提が表示だけの情報として埋もれたりする。

## 4. 実装フェーズ

### 小さく進める順序

1. 科目マスタと卒業要件判定を作る。
2. リスト表示で履修追加・削除・学期選択を安定させる。
3. CAP、要件外、資格認定、教職課程など例外ルールを足す。
4. ツリー表示用ノードとエッジを追加する。
5. リスト表示とツリー表示を同じ状態で同期させる。
6. UI の見やすさと操作性を調整する。
7. GitHub Pages で公開確認する。

### 実装時の注意

- 判定ロジックを UI ごとに複製しない。
- `state.planned` のような履修状態はリストとツリーで共有する。
- ツリー上のノードが同じ科目を複数コースに表示する場合、どのノードがコース必修表示を持つかを明確にする。
- 教職 ON/OFF のような一括操作は、ON 時の自動追加と OFF 時の保持方針を UI で伝える。
- 絞り込みと表示オプションは役割を分ける。重複するトグルは作らない。

## 5. UI 改善フェーズ

### 見やすさチェック

- 科目名が 1 行に収まるか。
- 長い科目名のフォントが小さくなりすぎていないか。
- ノードの高さが内容に合っているか。
- 矢印がノードに隠れていないか。
- 接続線の角度が急すぎないか。
- 同じ系列の I/II や演習 I/II が自然に並んでいるか。
- 上詰め、中央揃え、1 行レイアウトのルールが場当たり的になっていないか。
- ホバー情報が必要なノードだけに付いているか。
- 略称や要件用語はホバーで正式説明を確認できるか。
- リスト表示に不要なツリー専用 UI が出ていないか。

### 操作性チェック

- 履修ボタンを押した時だけ選択メニューが開くか。
- メニューが他ノードの下に隠れないか。
- 解除ボタンが科目情報と同じ行で扱いやすいか。
- 絞り込みメニューは必要な時だけ展開されるか。
- 標準配置、クリア、コース変更、教職課程 ON/OFF 後も状態が破綻しないか。

## 6. 検証チェック

### ローカル確認

- `node --check app.js`
- `git diff --check`
- ローカルサーバーで表示確認
- リスト表示とツリー表示の切り替え
- 絞り込みメニューの表示/非表示
- 標準配置の反映
- 資格取得と科目履修の排他
- 教職課程 ON/OFF
- コース未選択時とコース選択時の表示
- コンソールエラーなし

### GitHub Pages 確認

- Pages の参照ブランチを確認する。
- push 後に Pages build が `built` になったことを確認する。
- 公開 URL に `?v=<commit>` を付けてキャッシュを避ける。
- 公開 URL で `app.js` が最新コミットの内容になっているか確認する。
- ローカルと Pages で表示や操作がずれていないか確認する。

## 7. 開発環境と運用

### 推奨設定

- 静的サイトとして実装し、ビルド不要にする。
- ローカル確認は固定ポートを使う。
- ブラウザ確認用にキャッシュ回避クエリを使う。
- 資料 PDF、MD 化資料、実装コードを同じリポジトリで管理する。
- 作業ブランチは `codex/` プレフィックスにする。
- 安定版は `main`、試作版は `prototype/` ブランチとして退避する。

### よく使う確認コマンド

```sh
python3 -m http.server 8767
node --check app.js
git diff --check
git status --short --branch
gh api repos/kobashi/HyperCurriculumTree/pages
gh api repos/kobashi/HyperCurriculumTree/pages/builds/latest
```

### リリース前チェック

- README に現在版と公開 URL が書かれているか。
- CHANGELOG に今回の変更がまとまっているか。
- 公式資料との差分メモが更新されているか。
- 実装で直した内容が Markdown 資料にも戻っているか。
- Pages の参照ブランチが意図したブランチになっているか。
- 元の試作版を残す場合、退避ブランチが push 済みか。

## 8. 次回のための教訓

- PDF の見た目を一度読んだだけで実装に入らない。
- 公式資料、補助資料、実装資料の関係を先に決める。
- データの不足を見つけたら、アプリだけでなく資料にも反映する。
- UI の見やすさは後から必ず調整が必要になるため、ノード配置や接続線は変更しやすい形で作る。
- 「表示上の工夫」と「履修判定上のルール」を分ける。
- ユーザーが混乱した UI は、トグルやラベルを増やすより、役割を統合して減らす方がよい場合が多い。
- 完成後に、同じ手戻りを防ぐための振り返りドキュメントを残す。
